# SaPhARI parameter determination, validation, downstream search, and BLASTn support workflow

This directory contains the workflow used to derive, refine, validate, score, and apply SaPhARI family parameters from a curated set of satellite genomes.

The curated reference dataset used for rule generation consisted of known satellite genomes that were first annotated with SaPhARI against a DIAMOND database built from the RefSeq archaeal, bacterial, and viral protein snapshot dated **November 18, 2025**. These annotated reference genomes were used to derive candidate family-level rules, validate the resulting parameter set, and score recovery performance. Candidate rules generated by the extraction workflow were manually reviewed and refined to produce the finalized SaPhARI family rule file. The finalized rule set was subsequently applied to a larger collection of new genomes for downstream satellite discovery.

Because the full genome FASTA collection used for downstream searching is too large to retain in GitHub, the repository includes the curated satellite annotation archive, the downstream accession list, retained SaPhARI annotation/result outputs, and the scripts needed to document the analysis.

## Workflow summary

### 1. Annotated reference genomes

`annotated_satellites.zip` contains the curated satellite genome annotation dataset used for rule generation and validation. In the original local workflow, these inputs were organized by family and stored as DIAMOND-filtered TSV annotation files produced by SaPhARI. These annotated, family-organized reference files served as the input dataset for parameter extraction.

### 2. Parameter extraction and rule refinement

`extract_clusters.py` was used to infer candidate family rules from the annotated reference set. This step generated candidate marker sets and supporting diagnostic outputs written to `output/extraction/`, including:

* `family_rules_best.tsv`
* `family_rules_readable.txt`
* `cluster_backmap.tsv`
* `pairwise_confusion.tsv`
* `Families_postscript.json`

The candidate rules were then manually reviewed and refined based on family-level recovery, cross-family confusion, and biological interpretability. The finalized rule set used for validation, scoring, and downstream searching is retained as:

* `Families_final.json`

### 3. Validation

The finalized `Families_final.json` file was used to run SaPhARI back on the curated annotated reference set as a structured validation step. These validation runs are stored in `output/test_json/` as per-family output directories containing SaPhARI `RESULTS/` outputs.

### 4. Scoring

`score.py` was used to score the SaPhARI validation output against the expected curated family labels. This step generated per-file results, family-level summaries, and overall summary statistics for evaluating the finalized family rules.

### 5. Downstream search on new genomes

After validation and scoring, the finalized family rule set was applied to a larger collection of new genomes for downstream satellite discovery. The accessions for these genomes are listed in:

```text
all_accessions.txt
```

The original downstream genome FASTA files are not retained in this repository because of file size. Instead, the repository preserves the accession list and corresponding SaPhARI outputs. Downstream SaPhARI runs are stored in batched directories under `output/new_genomes_fasta/`, with each batch retaining:

```text
annotated/
RESULTS/
```

The `annotated/` folders contain the annotation-stage TSV outputs generated from the downstream genome inputs. The `RESULTS/` folders contain the family-level SaPhARI search outputs generated after applying the finalized parameter set.

### 6. Compilation of flagged downstream results

`compile_flagged_results.py` was used to compile non-empty downstream SaPhARI result files from `output/new_genomes_fasta/`. These compiled outputs are written to `output/compiled_flags/` and summarize flagged accessions, detected families, and multi-family calls.


### 7. Core-region BLASTn support analysis

A follow-up BLASTn support analysis was performed on the SaPhARI-called core regions from the downstream search. This analysis is stored in `blastn_scripts/` and compares predicted core regions against same-family satellite nucleotide reference databases.

The BLASTn workflow consists of:

* `01_extract_core_region_fastas.py` — extracts core-only nucleotide FASTAs from SaPhARI result TSVs.
* `02_prep_core_family_query_fastas.sh` — concatenates extracted core FASTAs into one query FASTA per family.
* `03_build_family_blastdb.sh` — builds same-family BLASTn nucleotide databases.
* `04_run_core_family_blastn.slurm` — runs the family-level BLASTn comparisons.
* `05_summarize_core_family_wide_support.py` — summarizes BLASTn support for each core query region.
* `06_compile_core_blastn_summary_metrics.py` — compiles expanded region- and family-level BLASTn support summaries.

The retained BLASTn outputs are stored in:

```text
output/blast_core_results/
output/blast_core_family_support/
```

The generated family-level query FASTA bundle directory, `output/blast_core_queries/`, is not retained because it can be regenerated from the extracted core regions and may be too large for GitHub.

## Database snapshot

The DIAMOND database used for SaPhARI annotation in this workflow was `SaPhARI_11_18_25.dmnd`, corresponding to a RefSeq archaeal, bacterial, and viral protein snapshot formatted for DIAMOND on **November 18, 2025**.

The BLASTn support workflow used family-specific nucleotide reference FASTA files to build local BLASTn databases. These generated database files are not retained in the repository.

## Reproducibility

The `slurm/` directory contains the batch scripts used to run parameter extraction, validation, scoring, and downstream genome searches on the cluster. The `blastn_scripts/` directory contains the scripts used for downstream core-region BLASTn support analysis.

Together with the archived reference dataset, accession list, retained annotation outputs, SaPhARI result outputs, compiled summaries, and specified DIAMOND database snapshot, these scripts document how the study was executed and provide the structure needed to recreate the analysis when the corresponding input datasets are available.

The repository intentionally excludes large genome FASTA inputs, runtime folders, generated BLAST query bundles, and database files. These files are either too large for GitHub or can be regenerated from the retained scripts, accession list, and source datasets.
